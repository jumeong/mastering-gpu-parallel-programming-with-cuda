# 49. Naive implementation of the matrix multiplication

## 1. 개요 및 중요성

* **컴퓨팅의 초석:** 시뮬레이션, 최적화, 선형 대수 연산의 핵심.
* **AI/DL의 엔진:** 신경망의 각 레이어는 본질적으로 행렬 연산이며, 이를 최적화하면 학습과 추론 속도가 직결됨.
* **주요 라이브러리:** * **NVIDIA:** cuBLAS(고성능), CUTLASS(유연한 커스텀).
* **AMD:** Tensile(설정 기반), rocBLAS.



## 2. 하드웨어 가속기 (Specialized Units)

행렬 연산만을 위해 설계된 전용 하드웨어 유닛을 사용하면 일반 코어 대비 수십 배의 처리량(Throughput)을 얻을 수 있습니다.

* **NVIDIA Tensor Cores:** A100 기준, 일반 코어 대비 약 10~20배 이상의 성능 향상.
* **AMD Matrix Cores:** CDNA 아키텍처 등에서 동일한 역할 수행.

## 3. CPU vs GPU 구현 차이

행렬 곱 $C = A \times B$ ($A: M \times K, B: K \times N, C: M \times N$) 기준:

| 구분 | CPU 구현 (Sequential) | GPU 구현 (Naive Parallel) |
| --- | --- | --- |
| **구조** | 3중 For 루프 (i, j, k) | 2D 스레드 인덱싱 + 1중 For 루프 (k) |
| **인덱싱** | `for(i...)`, `for(j...)` | `row = blockIdx.y * blockDim.y + threadIdx.y`, `col = blockIdx.x * blockDim.x + threadIdx.x` |
| **연산 방식** | 한 스레드가 모든 결과 값을 순차 계산 | 각 스레드가 결과 행렬 $C$의 요소 하나씩 할당받아 병렬 계산 |

## 4. CUDA 프로그래밍 7단계 워크플로우

1. **Memory Allocation:** `cudaMalloc`으로 GPU 메모리 확보.
2. **Initialization:** 호스트(CPU)에서 데이터 초기화.
3. **HtoD Copy:** `cudaMemcpy` (Host to Device)로 데이터 전송.
4. **Kernel Launch:** `<<<grid, block>>>` 설정으로 커널 실행.
5. **DtoH Copy:** `cudaMemcpy` (Device to Host)로 결과 회수.
6. **Validation:** CPU 결과와 GPU 결과를 비교하여 검증 (절대 오차 합 계산).
7. **Free Memory:** `cudaFree` 및 `delete`로 자원 해제.

## 5. 성능 분석 및 최적화 전략 (Profiling)

* **Nsight Compute 활용:**
  * **Duration:** 커널 실행 시간 측정 (Naive 방식은 오버헤드가 큼).
  * **Throughput:** Compute와 Memory의 사용률 확인.
  * **Roofline Analysis:** 현재 연산이 **Memory Bound**(메모리 대역폭 제한)인지 **Compute Bound**(연산 성능 제한)인지 판별.


* **Naive 방식의 한계:** 글로벌 메모리에 너무 빈번하게 접근함 ($2 \times K$번 읽기 당 2번 연산).
* **향후 과제 (Optimization):**
  * **Tiling (타일링):** 데이터를 작은 블록 단위로 쪼개어 처리.
  * **Shared Memory (공유 메모리):** 자주 쓰는 데이터를 L1 캐시급 속도의 공유 메모리에 올려 글로벌 메모리 접근 횟수를 획기적으로 줄임.

# 50. Optimize the Matrix Multiplication using the shared memory

## 1. 공유 메모리(Shared Memory) 할당

* **선언:** `__shared__ float s_a[TILE_SIZE][TILE_SIZE];`
* **목적:** 전역 메모리(Global Memory)에서 읽어온 타일 조각을 저장하여, 블록 내 스레드들이 데이터를 여러 번 재사용(Reuse)할 수 있게 함.
* **메모리 계산:** TILE_SIZE=16일 때, 한 블록당 약 2KB($16 \times 16 \times 4 \times 2$)를 사용하여 하드웨어 제한(보통 48KB~164KB) 내에서 안전하게 동작함.

## 2. 커널의 핵심 로직 (The Sliding Window)

타일링 연산은 결과값 $C(i, j)$를 계산하기 위해 행렬 $A$와 $B$를 **타일 단위로 훑으며(Iterate over tiles)** 진행됩니다.

1. **Index Mapping:** 전역 메모리 주소(`row`, `col`)를 계산하여 현재 스레드가 담당할 $C$의 위치를 확정.
2. **Tile Loop (K-dimension):** * **Load:** 전역 메모리에서 타일 조각을 가져와 `s_a`, `s_b`에 저장.
* **Sync:** `__syncthreads()`를 호출하여 블록 내 모든 스레드가 로드를 마칠 때까지 대기.
* **Compute:** 공유 메모리에 올라온 타일 데이터를 사용하여 부분 합(Partial Sum)을 계산.
* **Sync:** 다음 타일을 로드하기 전, 현재 타일 연산이 끝났음을 보장하기 위해 다시 동기화.


3. **Store:** 모든 타일에 대한 연산이 끝나면 최종 `sum`을 전역 메모리 $C$에 저장.

## 3. 성능 최적화 결과 (Benchmarks)

강의 내 실험 결과(1024x1024 기준)는 다음과 같습니다:

| 커널 종류 | 실행 시간 (approx.) | 성능 향상 |
| --- | --- | --- |
| **Naive Kernel** | ~3,000 $\mu s$ (3ms) | 기준점 |
| **Tiled Kernel** | ~2,000 $\mu s$ (2ms) | **약 33% 성능 향상** |

## 4. 하드웨어 제약 사항 (Critical Constraints)

강사가 강조한 **실패 사례(64x64 타일)**를 통해 배우는 CUDA 제약 조건:

* **스레드 블록 한계:** 하나의 블록은 최대 **1,024개**의 스레드만 가질 수 있음.
* **계산:** $32 \times 32 = 1,024$ (최대치), $64 \times 64 = 4,096$ (**오류 발생/실행 불가**).
* **권장 사항:** 보통 $16 \times 16$ (256개) 또는 $32 \times 32$ (최대한 활용 시)를 주로 사용하며, 연산 복잡도에 따라 최적의 `TILE_SIZE`를 찾아야 함.

# 51. Optimization of the MM using float4 (important)

## 🚀 CUDA 행렬 곱셈 최적화: `float4` & 벡터화 요약

강의의 핵심은 **"스레드 하나가 4개의 데이터를 한 번에 처리하게 만들어, 메모리 대역폭과 연산 효율을 동시에 잡는 것"**입니다.

### 1. 주요 최적화 기법: `float4`의 도입

* **개념:** 기존에는 스레드 하나가 `float` 데이터 1개를 읽었지만, 이제는 `float4`라는 내장 자료형을 사용해 **128-bit(4바이트 × 4)**를 한 번에 읽어옵니다.
* **효과:** * **메모리 대역폭 효율:** 메모리 요청 횟수가 줄어들어 메모리 바운드(Memory Bound) 문제가 해결됩니다.
* **연산 효율:** `sum[0]~sum[3]`처럼 루프를 풀어서(Unrolling) 작성함으로써 제어 오버헤드를 줄이고 연산 바운드(Compute Bound)를 개선합니다.


* **구조적 변화:** 스레드 수가 1/4로 줄어듭니다. 각 스레드가 4개 분량의 일을 하기 때문입니다.

## 2. 코드 레벨의 변화 및 주의점

| 구분 | 내용 |
| --- | --- |
| **인덱스 계산** | 각 스레드의 시작 주소에 `threadIdx.x * 4`를 곱해 겹치지 않게 오프셋을 설정합니다. |
| **데이터 로딩** | `reinterpret_cast<float4*>(...)`를 사용하여 전역 메모리의 주소를 `float4` 포인터로 강제 변환(Casting)해 한꺼번에 읽어옵니다. |
| **연산부** | `dot(.)` 연산자(`.x`, `.y`, `.z`, `.w`)를 사용하여 4개 성분의 부분합을 각각 계산합니다. |
| **동기화** | `__syncthreads()`를 통해 공유 메모리에 데이터 로딩이 완료된 후 연산을 시작하도록 보장합니다. |

## 3. 성능 개선 결과 (강의 내 벤치마크)

강의에서 보여준 실행 시간 변화는 매우 드라마틱합니다:

* **Naive (기본형):** 360 $\mu s$
* **Shared Memory (최적화 1단계):** 250 $\mu s$
* **Float4 + Unrolling (최종 최적화):** **150 $\mu s$** (약 2.4배 성능 향상)

---

## 💡 보충 설명: 왜 casting(`reinterpret_cast`)이 필요한가요?

강사님은 연구적 관점에서 **전역 메모리(Global)와 공유 메모리(Shared)의 주소 체계 차이**를 언급하셨습니다.

1. 기본적으로 행렬은 `float` 배열로 선언되어 있습니다.
2. 이를 `float4` 단위로 읽으려면 하드웨어에 "이 주소부터는 4개씩 묶인 데이터다"라고 알려줘야 합니다.
3. 이때 `reinterpret_cast`를 통해 데이터 타입의 해석 방식을 강제로 바꿔주는 것입니다.

# 52. Tiling in matrix multiplication

## 1. 타일링 결정의 3대 요소

성능을 결정짓는 가장 중요한 변수들은 다음과 같습니다.

* **행렬 크기 (Matrix Size):** 정사각형 행렬뿐만 아니라 비정형(Non-square) 행렬을 어떻게 다룰지가 관건입니다.
* **SM(Streaming Multiprocessor) 개수:** GPU의 핵심 연산 장치인 SM의 개수를 파악해야 합니다. 타일 수가 SM 개수보다 적거나 배수가 아니면 놀고 있는 SM이 생겨 자원이 낭비됩니다.
* **타일 크기 (Tile Size):** 공유 메모리(Shared Memory) 용량에 맞춰 적절한 크기를 정해야 하며, 반드시 정사각형일 필요는 없습니다 (예: 32x16).

## 2. 효율적인 타일링 예시 (SM 16개 기준)

강의에서 제시한 시나리오별 효율성 비교입니다.

| 시나리오 | 행렬 크기 | 타일 크기 | 총 타일 수 | 효율성 | 이유 |
| --- | --- | --- | --- | --- | --- |
| **최적** | 40x40 | 10x10 | **16개** | **최상 (Green)** | SM 16개에 딱 맞춰 100% 가동 |
| **비효율** | 40x50 | 10x10 | **20개** | **낮음** | 1회차(16개) 후 2회차에 4개만 작동 (12개 SM 유휴) |
| **최악** | 40x41 | 10x10 | **20개** | **매우 낮음** | 타일 수가 SM 배수도 아니고, 행렬 크기가 타일로 나누어떨어지지도 않음 |

## 3. 주요 최적화 팁

* **제로 패딩 (Zero Padding):** 행렬 크기가 타일 크기로 나누어떨어지지 않을 때, 0으로 채워진 행이나 열을 추가하여 크기를 맞추는 것이 해결책입니다.
* **SM 활용 극대화:** 타일의 총 개수는 반드시 **SM 개수의 배수**가 되도록 설계해야 합니다. (예: SM이 16개라면 타일은 16, 32, 48...개로 설정)
* **GTC 컨퍼런스 활용:** NVIDIA에서 주최하는 GTC(GPU Technology Conference)의 강연들을 참고하면 실무적인 병렬 프로그래밍 기법을 많이 배울 수 있습니다.

> **강사 코멘트:** "단순히 타일을 나누는 게 문제가 아니라, 공유 메모리에 타일이 들어가는지, 그리고 모든 SM이 동시에 일을 하고 있는지를 따지는 '스마트한 개발자'가 되어야 합니다."

53. The Tensor cores (introduction)

## 1. 텐서 코어(Tensor Core)란?
텐서 코어는 2017년 NVIDIA의 **Volta(볼타) 아키텍처**에서 처음 도입된 **행렬 연산(Matrix Multiplication) 가속기**입니다. 현대 딥러닝과 고성능 컴퓨팅(HPC)에서 가장 중요한 연산 단위로 평가받습니다.

* **주요 목적:** 딥러닝의 근간이 되는 행렬 곱셈 및 누산 연산을 비약적으로 빠르게 처리합니다.
* **AMD의 대항마:** AMD GPU에서는 이와 유사한 기능을 하는 **매트릭스 코어(Matrix Cores)**가 존재합니다.

---

## 2. 왜 텐서 코어가 중요한가? (성능 차이)
일반적인 CUDA 코어와 비교했을 때 텐서 코어는 압도적인 처리량을 자랑합니다.

* **연산 효율:** 일반 코어는 사이클당 **1개**의 연산을 수행하는 반면, 텐서 코어는 사이클당 **64개의 FMA(Fused Multiply-Add)** 연산을 수행할 수 있습니다.
* **FMA 연산이란?** 다음과 같은 공식을 한 번에 계산하는 방식입니다:
    $$D = C + (A \times B)$$
    * 곱셈과 덧셈이 포함되어 있어, 1개의 FMA는 **2개의 수학 연산(Ops)**으로 계산됩니다. 즉, 텐서 코어 하나당 사이클별로 **128개의 연산**이 가능한 셈입니다.
* **실제 사례:** Pascal(P100, 텐서 코어 없음) 대비 Volta(V100)는 약 **9배 이상의 성능 향상**을 보여주었습니다. (약 15 Teraflops → 125 Teraflops)



---

## 3. 아키텍처별 지원 데이터 타입
세대별로 텐서 코어가 다룰 수 있는 데이터 형식이 확장되었습니다.

| 아키텍처 | 도입 연도 | 주요 지원 데이터 타입 |
| :--- | :--- | :--- |
| **Volta** | 2017 | FP16, FP32 (정밀도 혼합) |
| **Turing** | - | INT8, INT4 추가 |
| **Ampere** | - | **TF32, BF16, FP64**, INT1 추가 |
| **Hopper / Blackwell** | - | 최신 데이터 타입 및 성능 고도화 |

> **Tip:** 입력값(A, B)은 낮은 정밀도(FP16)로 하되, 결과값(C, D)은 값의 손실을 막기 위해 높은 정밀도(FP32)로 저장하는 방식이 자주 쓰입니다.

---

## 4. 하드웨어 구조 (SM 내부)
GPU의 기본 단위인 **SM(Streaming Multiprocessor)** 내부에는 텐서 코어가 전략적으로 배치되어 있습니다. (V100 기준)

* **배치 구조:** 1개의 SM은 4개의 파티션으로 나뉘며, 각 파티션당 2개의 텐서 코어가 탑재됩니다.
* **총계:** 결과적으로 **1개의 SM당 8개의 텐서 코어**가 들어가며, V100 전체로는 총 640개의 텐서 코어가 존재합니다.



---

## 5. 소프트웨어 및 라이브러리 지원
NVIDIA는 하드웨어 성능을 끌어내기 위해 기존 라이브러리를 업데이트하고 프레임워크와 협력하고 있습니다.

* **라이브러리:** cuBLAS, cuDNN, cuSparse 등이 텐서 코어를 활용하도록 최적화되었습니다.
* **프레임워크:** PyTorch, TensorFlow, MXNet 등에서 행렬 연산을 실행하면 백그라운드에서 자동으로 텐서 코어를 사용하게 됩니다.
* **성능 최적화 팁:** 데이터 차원이 **128바이트의 배수**일 때 가장 효율적인 성능이 나옵니다.


# 54. Practical evaluation of the tensor cores


## 1. 실습 환경 및 조건
실험의 공정성을 위해 하드웨어 가속 방식(텐서 코어 사용 여부)을 제외한 모든 조건을 동일하게 설정했습니다.

* **연산 대상:** 정사각 행렬 곱셈 ($1024 \times 1024$ 및 $2048 \times 2048$)
* **데이터 정밀도:** 두 방식 모두 **반정밀도(Half Precision, FP16)** 사용
* **비교 대상:**
    1.  **Naive 버전:** 일반적인 방식의 행렬 곱셈 연산 (CUDA 코어 사용)
    2.  **Tensor Core 버전:** 텐서 코어 전용 프로그래밍 API 사용
* **테스트 장비:** NVIDIA RTX 3060 (Ampere 아키텍처)

---

## 2. 텐서 코어 코드 컴파일 및 트러블슈팅
텐서 코어 코드를 컴파일할 때는 단순히 `nvcc`를 사용하면 에러가 발생할 수 있습니다.

* **문제 발생:** `nvcc -o tensor tensor.cu` 실행 시 네임스페이스 관련 에러 발생.
* **원인:** 컴파일러가 기본적으로 텐서 코어가 없는 구형 아키텍처(예: Compute Capability 5.0 이하)를 기준으로 컴파일하려 하기 때문입니다.
* **해결책:** **컴퓨트 기능(Compute Capability)을 명시**해야 합니다.
    * 예: `nvcc -arch=sm_80 -o tensor tensor.cu` (Ampere 아키텍처 기준)
    * 텐서 코어는 Volta(sm_70) 이후 모델부터 지원됩니다.

# 55. Programming the Tensor cores using WMMA APIs

## 1. WMMA API 개요 및 하드웨어 요구사항
WMMA는 NVIDIA GPU의 **Tensor Core**를 직접 제어하여 행렬 연산(Matrix Multiplication)을 가속화하는 API입니다.

* **지원 아키텍처:** Compute Capability 7.0 이상 (**Volta** 아키텍처 이후부터 지원).
* **이유:** Tensor Core 하드웨어가 Volta 아키텍처에서 처음 도입되었기 때문입니다. (그 이전 모델인 Pascal 등에서는 실행 불가)
* **연산 단위:** 기존 CUDA 커널이 **Thread/Block** 단위로 연산했다면, WMMA는 **Warp(32개 스레드 집합)** 단위로 연산을 수행합니다.

## 2. WMMA의 4단계 매커니즘
강의에서는 표준 행렬 곱셈 알고리즘의 4단계를 WMMA에 맞춰 재해석합니다.

| 단계 | 설명 | WMMA 주요 특징 |
| :--- | :--- | :--- |
| **1. Partitioning (Tiling)** | 큰 행렬을 작은 타일로 나눔 | **Warp** 단위로 타일을 할당하며, 고정된 타일 사이즈(예: $16 \times 16 \times 16$)를 사용함. |
| **2. Loading** | 메모리에서 데이터를 가져옴 | `load_matrix_sync`를 사용. 공유 메모리가 아닌 **레지스터(Fragments)**로 데이터를 직접 로드함. |
| **3. Execution (MMA)** | 실제 행렬 곱셈 수행 | `mma_sync` 함수 호출. Tensor Core가 하드웨어적으로 $D = A \times B + C$를 계산. |
| **4. Storing** | 결과를 전역 메모리에 저장 | `store_matrix_sync`를 사용하여 레지스터에 있는 연산 결과를 전역 메모리로 전송. |

## 3. 핵심 개념: Fragment & Layout
* **Fragment:** 타일 데이터가 Warp 내 스레드들의 레지스터에 분산 저장된 형태를 의미합니다. 사용자는 내부 구조를 직접 관리할 필요 없이 API를 통해 접근합니다.
* **Matrix Layout:** `row_major` 혹은 `col_major`를 설정합니다. 성능 최적화를 위해 A 행렬은 행(Row) 방향으로, B 행렬은 열(Column) 방향으로 로드하여 메모리 접근 효율(Coalescing)을 높이는 기법이 강조되었습니다.



## 4. Warp ID 계산 (가장 중요한 구현 디테일)
WMMA는 Warp 단위 연산이므로, 각 Warp가 행렬의 어느 타일을 담당할지 결정하는 **Warp ID** 계산이 필수적입니다. CUDA는 `threadIdx`, `blockIdx`는 제공하지만 `warpId`는 기본 변수로 제공하지 않기 때문에 직접 계산해야 합니다.

* **기본 공식 (1D Block):**
    $$WarpID = \frac{threadIdx.x}{32}$$
* **다차원 그리드 확장:**
    그리드 내 전체 Warp 인덱스를 구하기 위해 `blockIdx`와 `blockDim`을 조합하여 계산합니다. 계산된 `WarpID`를 행렬의 행($m$)과 열($n$) 인덱스로 맵핑하여 각 Warp가 독립적인 타일을 처리하도록 설계합니다.
